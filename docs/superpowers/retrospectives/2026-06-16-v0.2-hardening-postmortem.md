# v0.2 Hardening — Failure Patterns Postmortem

**Date:** 2026-06-16
**Tags involved:** v0.2.0 → v0.2.1
**Hardening rounds:** R19 (red-team) → R20 (e2e) → R21 (xval) → R22 (final)
**Net result:** 23/23 acceptance → 0/8 red-team → 103/104 final = 99%

The v0.2.0 formal benchmark (R15-R18) passed 23/23 = 100%. The red-team round (R19) immediately found 0/8 — a complete collapse — because the formal benchmark cases were authored alongside the spec and shared the same blind spots. Subsequent hardening exposed 6 categories of spec failure that recurred enough to be considered systemic. This document records the patterns so future skill design can avoid them.

## The 6 systemic failure patterns

### Pattern ① — Repeat-run blindness (Idempotency)

**What we did wrong:** Wrote spec assuming first-time execution. Common cases:
- Manifest update used `replace "single" → "multi-agent"` string-match. Worked once; second run found no `"single"` and silently failed.
- `_intake/<role>.md` was opened in append mode by specialists. A leftover scratch file from a prior interrupted run mixed with fresh output and contaminated aggregation.
- `findings.md` was written unconditionally. A user-edited or prior-intake `findings.md` was silently overwritten.

**Diagnostic question to apply during spec writing:**
> "If I run this exact action twice — or more — what state remains? Is the second run an identity operation, an error, or a new initialization?"

If the answer isn't one of those three, the spec has an idempotency gap.

**Fix pattern used:** Unconditional set instead of conditional replace; archive-then-clear instead of append; pre-existing-file check before any overwrite.

---

### Pattern ② — Contract described but not validated

**What we did wrong:** The dispatch template told specialists "write to `_intake/<role>.md`", and the aggregate step read those files — but no step verified that specialists actually obeyed. A specialist that wrote to `_intake/insight-hunter.md` (full role name instead of short name) silently lost its entire output, and the coordinator reported a clean run.

**Diagnostic question:**
> "For every contract clause I told a subagent / external actor to obey, where in MY code does verification happen that it actually obeyed? If the answer is 'I trust the contract,' that's a gap."

**Fix pattern used:** Post-dispatch validation step that checks file presence, contents shape, and ID prefix conformance. Contract violations get logged to `_intake/_violations.md` and treated as `Found: 0`.

---

### Pattern ③ — Edge cases in cascading rules

**What we did wrong:** Wrote rules for the typical case and stopped. Examples:
- Cascade demotion handled the single-parent case (`🧪 references one parent`); didn't say what to do for multi-parent.
- Pair-check TODO emitted `Need experiment for I-<id>` even when `I-<id>` itself was demoted to Unverified — false positive.
- Stable ID collision rule said "in the same section" — missed cross-prefix collisions readable as identical hex.

**Diagnostic question:**
> "For every rule that operates on a set of items, what happens at: 0 items, 1 item, 2 items, N items, items with repeated keys, items pointing at each other?"

**Fix pattern used:** Explicit cardinality clauses. For every rule, write the 0-N behavior table.

---

### Pattern ④ — Cross-document drift

**What we did wrong:** Edited one spec file without propagating to siblings. Example: introduced `_intake/_prior/` and `_intake/_violations.md` in `deep-research/SKILL.md` during R19 fixes, but didn't add them to `workspace-spec.md`'s file inventory table. R21 cross-validation caught it.

**Diagnostic question:**
> "What is the canonical place for this information? Is every other place that references it consistent? When I changed the canonical place, did I propagate?"

**Fix pattern used:** On every spec edit, grep the repo for the changed concept and reconcile.

---

### Pattern ⑤ — Thin recovery paths

**What we did wrong:** Specified happy path + a single failure mode (specialist returns Found:0). Didn't systematically design for recovery from arbitrary mid-states:
- Interrupted intake leaving inconsistent manifest.
- User manually deleting `_intake/` (we explicitly told them they could after a week).
- Multiple workspaces in same cwd.
- User pre-editing `findings.md` before intake fires.

Each of these is a real-user state that the spec didn't address until benchmark found it.

**Diagnostic question:**
> "If a user walks into this workspace mid-stream — interrupted, manually edited, partially deleted, week-old — does my spec produce a sensible result, or does it assume the file system reflects exactly the last action my skill took?"

**Fix pattern used:** For every persistent artifact, design what to do when it's: missing, present-but-stale, present-but-malformed, present-and-conflicting-with-new-state.

---

### Pattern ⑥ — Loose natural-language enumeration

**What we did wrong:** Used loose English where strict enumeration would have caught more.
- Resume override listed `"继续主题 Y"` but not `"回到 X"` / `"切回 X"` / `"resume X"`.
- "Cosine-similar wording" for dedup — too vague to apply uniformly.
- "In the same section" for ID collision — interpreted literally, missed cross-prefix.

**Diagnostic question:**
> "If I were the model running this rule, can I list every phrase that triggers it? Every criterion that satisfies the comparison? Or does this require subjective judgment?"

**Fix pattern used:** Enumerate concrete trigger phrases. For comparisons, give 3+ concrete criteria with OR logic.

---

## Meta-lesson: the spec writer is biased toward happy path

All 6 patterns share one root cause: when writing spec, the writer pictures a successful run with cooperative inputs. The "fail" half of the behavior space is invented *reactively* during benchmark rounds.

The remedy is procedural — apply [`docs/superpowers/checklists/skill-spec-quality-checklist.md`](../checklists/skill-spec-quality-checklist.md) **before** running any benchmark. If the spec passes those 6 categories deliberately, the red-team round becomes a confirmation pass rather than a 0/8 collapse.

## Cost of catching late vs early

| Phase | Cost per gap caught |
|---|---|
| During spec brainstorming | ~1 minute (add one sentence) |
| During plan writing | ~5 minutes (revise plan + spec) |
| During formal benchmark (R15-R18) | ~10 minutes (small fix + re-round) |
| During red-team (R19) | ~30 minutes (substantial spec edit + re-round + re-xval) |
| In production (real user) | unknown — potentially data loss / silent failure |

The hardening rounds cost roughly 4× more than equivalent gaps caught during brainstorming. Applying the checklist upfront is the cheapest hardening available.
