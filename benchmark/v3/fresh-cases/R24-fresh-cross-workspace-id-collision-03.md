# R24-fresh-cross-workspace-id-collision-03

**Surface:** Stable ID reuse across independent workspaces — user asks about "I-a3f2c1" while in workspace B  
**Round:** 24  
**Category:** ⑥ (spec ambiguity / underspecification)  
**Not previously tested:** R19 (RT-V2-STABLE-ID-HASH-COLLISION-07) tested two findings *within the same workspace* colliding on the same ID. R13/R21 tested cross-prefix collision (I-a3f2c1 vs B-a3f2c1). No prior case tested the same 6-hex ID appearing in two *different* workspaces with different meanings.

---

## Precondition

Two workspaces exist under `.deeptutor/`:

**Workspace A** (`.deeptutor/attention-mechanism/`):
- `findings.md` contains `- [ ] **I-a3f2c1** Scaled dot-product omits temperature — [attn.py:142-158](...)`

**Workspace B** (`.deeptutor/lora-finetuning/`):
- `findings.md` contains `- [ ] **I-a3f2c1** Low-rank adapter rank r is hard-coded in config — [adapter.py:37-44](...)`

Both were generated by independent intakes. The SHA1 hash collision across different topics is coincidental but possible (6-hex = ~16M values; across 2 workspaces with ~10 findings each, probability is low but non-zero in a large user corpus).

The user is currently in Workspace B (last turn used `.deeptutor/lora-finetuning/`).

---

## Stimulus

User message (turn 4, in workspace B session):
> "我想深入讨论一下 I-a3f2c1 这个发现。"

---

## Expected behavior (per spec)

Per `heavy-mode.md` §Phase 1 §Choose ONE action (item a): "pick an unchecked `[ ]` item from `findings.md` related to the current `learning_path` node."

Per `SKILL.md` §Natural-language topic-switch detection §condition (c): "the message does NOT cite any item in the **current workspace's `findings.md`** (by stable id or paraphrase). When multiple workspaces exist under `.deeptutor/` in the same cwd, **only the active workspace's `findings.md` matters for this check.**"

So the spec explicitly says: when the user cites a stable ID, use the *current workspace's* `findings.md`. In this case, the current workspace is B, and `I-a3f2c1` exists in B's `findings.md` — so it should be treated as a reference to B's finding (low-rank adapter rank).

**Minimum bar to PASS:**
1. The coordinator MUST use Workspace B's `findings.md` to resolve `I-a3f2c1`.
2. It MUST discuss the LoRA finding (low-rank adapter rank), NOT the attention mechanism finding.
3. It MUST NOT surface Workspace A's finding for `I-a3f2c1`.
4. It MUST NOT ask for disambiguation between two workspaces when the ID exists in the current workspace.

---

## Simulation

**Step 1:** User is in Workspace B. Turn-type dispatch (SKILL.md §Turn-type dispatch): "Turn 2+ — you already have a workspace loaded: SKIP Step 1 entirely... read manifest.yaml for the persisted entry_mode / intent / current_mode."

**Step 2:** Check user-overrides — none match. Go to Phase 1 heavy-mode loop.

**Step 3:** Phase 1 §Read state: scan `findings.md` for unchecked `[ ]` items. The current workspace is B. `I-a3f2c1` in B's findings.md is found.

**Step 4:** User explicitly references `I-a3f2c1` — the NL topic-switch condition (c) blocks topic-switch detection because the ID exists in the active workspace. Correct behavior per spec.

**Step 5 (edge — where ambiguity bites):** The spec says the coordinator USES the current workspace's findings.md. But does it also inadvertently read Workspace A? The spec says "NEVER write outside `.deeptutor/<slug>/`" (SKILL.md §Workspace contract) but does NOT explicitly prohibit READING from sibling workspaces. The only "only active workspace" disambiguation is in the NL topic-switch condition (c) — it is not restated as a general read-isolation rule.

A coordinator that does a global scan of `.deeptutor/*/findings.md` (e.g., to do cross-workspace orphan detection from R23 fix) could accidentally surface Workspace A's finding for the same ID.

**Verdict: UNCLEAR** (spec's intent is clear for the NL-topic-switch check, but general read isolation across workspaces is not formally stated)

**Failure classification: ⑥** (spec underspecification — read isolation for sibling workspaces is implied but not explicit)

**Key gap:** The spec says "only the active workspace's findings.md matters" only in the context of the NL topic-switch check (condition c). It does not state this as a global read-isolation principle. A pedantic implementation that scans all `.deeptutor/*/findings.md` for the ID and finds it in two workspaces would have no spec guidance on how to proceed.

---

## Recommended fix (for R25)

Add a general read-isolation principle to `workspace-spec.md` or `SKILL.md §Workspace contract`:
> "When resolving any stable ID reference (in user messages, quizzes, or learning_log), look up only the *active workspace's* `findings.md`. Sibling workspace files are never read during an active session unless the user explicitly invokes a cross-workspace command (e.g., 'resume X')."
